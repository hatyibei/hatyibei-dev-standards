# ADR-002: テスト駆動開発の強制

## ステータス
accepted

## コンテキスト

AI生成コードは「動くが正しくない」状態になりやすい。テストなしの実装は検証不可能で、リグレッションの温床となる。

## 決定

RED-GREEN-REFACTOR サイクルを開発の基本単位とする。

### ワークフロー

```
1. RED    - 失敗するテストを先に書く
2. GREEN  - テストを通す最小限の実装
3. REFACTOR - テストが通る状態を維持しながら改善
```

### ルール

1. **テストファースト**: 実装コードの前にテストを書く
2. **最小実装**: テストを通すために必要最小限のコードだけ書く
3. **カバレッジ基準**: 80%以上を目標とする
4. **統合テスト優先**: モックよりも実際の依存関係を使うテストを優先
5. **検証してから完了宣言**: テストが全てパスしたことを確認してからタスク完了

### アンチパターン (禁止事項)

- テストなしでの機能追加
- テストとプロダクションコードの同時コミット（TDDの場合はRED→GREENの順序を記録）
- `--no-verify` によるテストスキップ
- モックの過剰使用（前四半期のモック/本番乖離インシデントの教訓）

## 根拠

- **obra/superpowers**: TDDスキルが中核、RED-GREEN-REFACTORの厳格な実施
- **affaan-m/everything-claude-code**: 80%+カバレッジ要件、TDDワークフロー必須
- **awslabs/aidlc-workflows**: Construction phaseにテスト検証を組み込み

## 影響

- 初期開発速度は若干低下するが、長期的な品質と保守性が向上
- AI生成コードの正しさを客観的に検証可能
- リファクタリングの安全ネットとして機能
