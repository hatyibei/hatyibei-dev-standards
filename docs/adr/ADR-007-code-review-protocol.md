# ADR-007: コードレビュープロトコル

## ステータス
accepted

## コンテキスト

AI生成コードは人間のレビューなしにマージすべきではない。構造化されたレビュープロトコルにより、品質の一貫性と見落としの防止を図る。

## 決定

2段階レビュープロトコルを採用する。

### 第1段階: 仕様準拠レビュー

- [ ] 要件・仕様を満たしているか
- [ ] エッジケースが考慮されているか
- [ ] テストが要件をカバーしているか
- [ ] 破壊的変更がないか

### 第2段階: コード品質レビュー

- [ ] セキュリティ脆弱性がないか (OWASP Top 10)
- [ ] パフォーマンスの問題がないか
- [ ] 命名規則・コーディング規約に従っているか
- [ ] 不要な複雑さがないか (YAGNI)
- [ ] エラーハンドリングが適切か

### レビュー原則

1. **仕様準拠を先に確認**: 品質が高くても仕様を満たさなければ不合格
2. **完全なdiffをレビュー**: 最新コミットだけでなく、PR全体の差分を確認
3. **Human Partner レビュー**: AI レビュー後に必ず人間がレビュー
4. **1 PR = 1 問題**: PRは単一の問題を解決する

### 自動拒否条件

- テストが失敗している
- セキュリティ脆弱性が検出された
- `.env` や認証情報の混入
- 未使用の依存関係の追加

## 根拠

- **obra/superpowers**: 2段階レビュー (仕様準拠 → 品質)、code-reviewer エージェント
- **affaan-m/everything-claude-code**: security-reviewer、language-specific reviewer
- **awslabs/aidlc-workflows**: 品質ゲート、監査ログ

## 影響

- レビュープロセスの標準化で品質が安定
- レビュー時間の増加（ただし手戻りの減少で相殺）
- AIと人間の役割分担が明確化
