# ADR-008: 品質ゲートと適応的深度

## ステータス
accepted

## コンテキスト

全てのタスクに同じレベルの厳密さを適用するのは非効率。タスクの複雑さに応じてプロセスの深度を調整しつつ、品質の下限は維持したい。

## 決定

適応的深度モデルと品質ゲートを組み合わせる。

### 深度レベル

| レベル | 対象 | プロセス |
|-------|------|---------|
| **minimal** | typo修正、設定変更、ドキュメント更新 | 直接修正 → テスト → コミット |
| **standard** | 機能追加、バグ修正、リファクタリング | 計画 → TDD → レビュー → コミット |
| **comprehensive** | アーキテクチャ変更、新規プロジェクト、破壊的変更 | ブレスト → 設計 → 計画 → TDD → 2段階レビュー → コミット |

### 品質ゲート (全レベル共通)

以下は深度レベルに関わらず必ず通過する:

1. **テスト通過**: 既存テストが全てパスすること
2. **セキュリティ**: 認証情報の混入なし、既知の脆弱性なし
3. **ビルド成功**: プロジェクトが正常にビルドできること
4. **検証**: 変更が意図通りに動作することの確認

### ゲート通過失敗時

- ゲートで失敗した場合、先に進まず原因を調査・修正
- `--no-verify` 等によるゲートの迂回は禁止
- 3回連続で失敗した場合はユーザーにエスカレーション

### コンテキスト管理

- 各ゲート通過時にコンテキストをリセット可能
- チェックボックスで進捗を追跡
- 全ユーザー入力を監査ログに記録（タイムスタンプ付き）

## 根拠

- **awslabs/aidlc-workflows**: 適応的深度（minimal/standard/comprehensive）、品質ゲート、監査ログ
- **obra/superpowers**: verification-before-completion スキル、エビデンス主義
- **affaan-m/everything-claude-code**: 80%+カバレッジ要件、セキュリティファースト

## 影響

- 単純タスクのオーバーヘッド最小化
- 複雑タスクの品質担保
- 深度判定の誤りリスク（ユーザーがオーバーライド可能）
