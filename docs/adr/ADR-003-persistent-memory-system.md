# ADR-003: 永続メモリシステム

## ステータス
accepted

## コンテキスト

Claude Code のセッションはコンテキストウィンドウに制限される。セッション間でプロジェクト知識、ユーザー嗜好、過去の判断を保持する仕組みが必要。

## 決定

ファイルベースのメモリシステムを採用し、claude-mem の設計思想を参考にした軽量な永続化を行う。

### メモリ構造

```
~/.claude/projects/<project>/memory/
  ├── MEMORY.md          # インデックス (200行以内)
  ├── user_*.md          # ユーザー情報
  ├── feedback_*.md      # フィードバック・行動指針
  ├── project_*.md       # プロジェクト状態
  └── reference_*.md     # 外部リソース参照
```

### 各メモリファイルの形式

```markdown
---
name: メモリ名
description: 一行説明（将来の検索用）
type: user | feedback | project | reference
---

内容（feedback/projectはルール → Why → How to apply の構造）
```

### 原則

1. **コードから導出可能な情報は保存しない**: ファイルパス、アーキテクチャ、git履歴は保存不要
2. **重複禁止**: 新規作成前に既存メモリの更新を検討
3. **鮮度管理**: 相対日付は絶対日付に変換して保存
4. **200行制限**: MEMORY.mdは200行以内に収める
5. **検証してから参照**: メモリの情報は現在のコード状態と照合してから使用

### claude-mem からの着想（高度な運用時）

- セッション要約の自動生成
- 観測の重複排除（コンテンツハッシュベース）
- プログレッシブ・ディスクロージャ（インデックス → タイムライン → 詳細の3層）

## 根拠

- **thedotmack/claude-mem**: SQLite + ベクトルDB による高度なメモリ管理、セッション間知識保持
- **affaan-m/everything-claude-code**: `~/.claude/skills/learned/` の学習スキル管理
- Claude Code 組み込みのファイルベースメモリシステム

## 影響

- セッション間の文脈連続性が向上
- メモリの陳腐化リスクがあるため定期的な棚卸しが必要
- 高度な運用（ベクトル検索等）は必要に応じて段階的に導入
