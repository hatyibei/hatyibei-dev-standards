# ADR-001: AI駆動開発ライフサイクル

## ステータス
accepted

## コンテキスト

AI コーディングエージェントを活用する開発では、従来のウォーターフォールやアジャイルとは異なるライフサイクルが必要になる。闇雲にコード生成を始めるのではなく、構造化されたフェーズを経ることで品質と再現性を担保したい。

## 決定

AIDLC (AI-Driven Development Life Cycle) の3フェーズモデルを採用し、Superpowers のワークフロー進行を統合する。

### フェーズ構成

```
INCEPTION (何を・なぜ作るか)
  ├── ワークスペース検出
  ├── リバースエンジニアリング (既存コードベースの場合)
  ├── 要件分析 (適応的深度)
  ├── ブレインストーミング (Socratic対話)
  └── ワークフロー計画 (bite-sizeタスク分解)

CONSTRUCTION (どう作るか)
  ├── 設計 → テスト → 実装 (TDDサイクル)
  ├── サブエージェント駆動の並列実行
  ├── ビルド & テスト検証
  └── コードレビュー (2段階)

OPERATIONS (運用)
  ├── デプロイ検証
  ├── モニタリング
  └── 振り返り & メモリ保存
```

### 適応的深度

タスクの複雑さに応じて各ステージの深度を調整する:
- **minimal**: バグ修正、typo修正 → 計画スキップ可
- **standard**: 機能追加、リファクタリング → 標準フロー
- **comprehensive**: アーキテクチャ変更、新規プロジェクト → 全ステージ詳細実行

### Human-In-The-Loop

重要な判断ポイントでは必ずユーザー確認を挟む:
- 要件の解釈確認
- 設計方針の承認
- 破壊的変更の実行前

## 根拠

- **awslabs/aidlc-workflows**: 3フェーズモデル、適応的深度、Human-In-The-Loop設計
- **obra/superpowers**: brainstorming → plan → execute の段階的ワークフロー
- **affaan-m/everything-claude-code**: plan-before-execute の原則

## 影響

- 単純なタスクに対するオーバーヘッドは適応的深度で最小化
- ライフサイクルの各フェーズで成果物(ドキュメント)が生成される
- ユーザーがフェーズをスキップする自由は維持する
